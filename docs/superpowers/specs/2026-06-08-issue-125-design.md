# Issue #125: Incorrect Payload via HTTP Sync

## Problem Statement

When the Dart-side custom `buildSyncBody` callback times out or throws an error, the native Tracelet SDKs incorrectly handle the failure:
- **iOS (`TraceletIosPlugin.swift`)**: Explicitly returns the literal string `{"error": "TIMEOUT"}` or `{"error": "INVALID_TYPE"}`. The native sync manager treats this as a valid custom JSON payload and POSTs it to the backend.
- **Android (`TraceletAndroidPlugin.kt`)**: Returns `null` on timeout. However, `NativeSyncProvider.kt` interprets `null` as "no custom body provided" and falls back to sending the **default** Tracelet JSON array payload.

Both behaviors are incorrect because a backend expecting a specific custom telemetry payload will fail to parse these unexpected formats, resulting in HTTP 400 errors or internal server errors on the backend.

## Goal

Ensure that when a custom sync body generation fails or times out, the SDK securely **aborts the sync attempt** and retries later, completely preventing the transmission of unexpected fallback payloads.

## Proposed Changes

### iOS Changes
1. **`TraceletIosPlugin.swift`**: 
   - Update `requestSyncBody(locations:)` to return `nil` instead of literal JSON error strings (`{"error": "TIMEOUT"}`, `{"error": "INVALID_TYPE"}`, `{"error": "FLUTTER_ERROR"}`).
   - *Result*: The interceptor will return `nil` to the iOS `TraceletSyncPlugin.swift` and `NativeSyncProvider.swift` equivalents.

2. **`TraceletSyncPlugin.swift`** (in `packages/tracelet_sync/ios/...`):
   - Check if the dart callback is registered to provide a custom body.
   - If the custom body builder fails (returns `nil` or errors out), **return 0** to abort the sync instead of falling through to the default `syncManager.syncBatchBlocking`.

### Android Changes
1. **`NativeSyncProvider.kt`** and **`TraceletSyncPlugin.kt`**:
   - Currently: `if (customBody != null) { ... } return syncManager.syncBatchBlocking(...)`
   - Proposed Fix: We need to determine if a custom builder is *expected*. If the user configured a custom body builder, but it returns `null` (due to timeout/error), we must **return 0L** instead of falling through to the default sync block. We will add a check to verify if the Dart side has a registered builder.

### Example App
1. **`example/lib/issues_page.dart`**:
   - Add a "Green Test: Issue #125" button.
   - The button configures the HTTP sync with a custom body builder that intentionally delays beyond the timeout (e.g., `Future.delayed(Duration(seconds: 15))`).
   - Verifies via the UI that no network request payload containing `{"error": "TIMEOUT"}` or the default fallback was sent.

## Verification & Testing Plan
- **Unit Tests**: Update the iOS `TraceletIosPluginTests` to verify `nil` is returned on timeout instead of `{"error": "TIMEOUT"}`.
- **Integration Tests**: Write an integration test for both platforms validating that if `buildSyncBody` times out, the sync attempt is aborted and the `http_events` log does not contain a fallback payload.

## Open Questions for Review
- Since Dart `MethodChannel` calls from Android/iOS currently just return `null` when they time out, we need a way to distinguish between "No custom builder registered" (so use default payload) vs "Custom builder registered but timed out" (so abort). Are we okay with updating the `DartSyncInterceptor` interface or Dart channel to explicitly return a distinct error state versus `null`?
